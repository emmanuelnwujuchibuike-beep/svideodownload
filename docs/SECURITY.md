# Security posture

How Frenz defends against the common web attack classes, what was audited on
2026-07-06, and the rules that keep it that way. No system is "immune" — this
is defense-in-depth: several independent layers each have to fail before an
attacker gets anywhere.

## SQL injection — not reachable by construction

- **No raw SQL exists anywhere in the app.** Every database access goes
  through the Supabase client (PostgREST), which builds parameterized queries;
  user input is never concatenated into SQL text.
- The only stored procedures called (`.rpc(...)`) are named Postgres functions
  with typed arguments (`bump_post_counter`, `recompute_trust_score`,
  `recompute_hot_score`) — arguments are bound server-side, never interpolated.
- **Row Level Security (RLS)** is enabled on user-facing tables (see
  `supabase/migrations/`) — even a bug in an API route can't read/write rows
  the policies don't allow, because the anon/user key is policy-bound.
- The service-role client (`lib/supabase/admin.ts`) bypasses RLS by design and
  is therefore **server-only**: it reads `SUPABASE_SERVICE_ROLE_KEY`, which is
  never shipped to the browser (not `NEXT_PUBLIC_`), and every route using it
  authenticates the caller first.
- **Rule:** never use string interpolation to build a query or RPC; new tables
  get RLS policies in the same migration that creates them.

## XSS

- React escapes all interpolated text by default; user content (titles, bios,
  captions, comments) renders through React elements — never `innerHTML`.
- **JSON-LD structured data** was the one real gap (fixed 2026-07-06):
  `JSON.stringify` doesn't escape `<`, so user-controlled strings inside
  `<script type="application/ld+json">` could break out of the tag. All
  user-content pages now serialize via `lib/seo/json-ld.ts` (`jsonLd()`),
  which escapes `< >` `&` to `\uXXXX`.
- **Rule:** any new `<script dangerouslySetInnerHTML>` with non-constant data
  MUST use `jsonLd()`; `dangerouslySetInnerHTML` for anything else needs
  review.

## CSRF

- Sessions are Supabase httpOnly cookies with `SameSite=Lax` — cross-site
  pages can't ride them on the POST/PATCH/DELETE requests that change state.
- State-changing endpoints require JSON bodies (`Content-Type:
  application/json`), which plain cross-site HTML forms cannot send.
- Native/worker clients authenticate with explicit `Authorization: Bearer`
  headers instead of ambient cookies.
- **Rule:** never mutate state on GET.

## Secrets

- `.env.local` is gitignored; the full git history was scanned on 2026-07-06
  for key patterns (Resend `re_…`, Supabase JWTs, SMTP credentials, generic
  `password=`) — **zero real credentials have ever been committed**.
- The GitHub/GitGuardian "SMTP credentials visible" alert matched an
  *instructional* Host/Username/Password block in `docs/EMAIL_SETUP.md` that
  contained only placeholder text; the block was rewritten so it no longer
  looks like a credential dump. If an alert ever names a specific token,
  rotate it at the provider and update Vercel — rotation is always cheap.
- **Rule:** secrets live in Vercel/Railway env vars and `.env.local` only.
  Docs never show values, and never use `key: value` credential formatting
  even for placeholders.

## Authentication

- Passwordless Email OTP: codes are generated by Supabase Auth (bcrypt-hashed
  at rest, single-use, expiring) and verified with `verifyOtp` — the app never
  stores or logs codes; the send endpoint is rate-limited per-IP AND per-email
  and its responses don't reveal whether an account exists.
- OAuth (Google) goes through Supabase's server-side code exchange.
- Session tokens are httpOnly cookies (unreadable from JS), auto-refreshed by
  Supabase; the middleware revalidates on navigation.

## Platform layers

- **Headers** (next.config.ts, all routes): `X-Content-Type-Options: nosniff`,
  `X-Frame-Options: SAMEORIGIN` (clickjacking), `Referrer-Policy:
  strict-origin-when-cross-origin`, `Permissions-Policy` (camera/mic/geo off),
  `Strict-Transport-Security` (2y, preload). HTTPS is enforced by Vercel.
- **Rate limiting** (`lib/rate-limit.ts`): Upstash sliding windows on
  metadata/downloads/messages/OTP + daily caps; OTP tightest at 4/min.
- **Input validation:** API routes validate with zod (or strict regex — UUID
  params) before touching data.
- **Admin/debug surfaces** (`/api/admin/*`) require the `x-worker-secret`
  header matching `WORKER_SECRET`.
- **Uploads** go to object storage (R2/Supabase Storage) under generated keys
  — user bytes are never written into the app's filesystem or executed.
